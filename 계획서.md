# 📋 계획서 — Auto-Pilot v2.0

## 목표
다중 스텝 입력 UI + 완료 토큰 감지(템플릿·OCR) + 사전 검증으로 구성된 스텝 기반 자율 루프.
클래식 모드(SMART_PROMPT)와 공존하며, 텔레그램 양방향 제어·F12 긴급 정지를 갖춘다.

## v2.0 아키텍처 요약

### 상태 머신
```
IDLE → MONITORING ←→ GENERATING
          ↓               ↓ (done)
       WAITING        HANDLE_DONE
          ↓               ├─ 스텝 완료? → expected_step++ → RESUMING
       RESUMING        ├─ 계속 필요? → continue_count++ → RESUMING / PAUSED
          ↑            └─ 클래식: LOOPSTABLE? → MONITORING
          └───────────────────────────────────────────────────┘
```

### 스텝 완료 감지 (이미지 매칭 전용)
1. 프롬프트가 모델에게 응답 끝에 `[STEP{N}_DONE]` 표식을 출력하도록 지시
2. 사용자가 그 표식을 캡처해 `step{N}_complete.png` 로 저장
3. `step{N}_complete*.png` 를 `ready_confidence` 로 이미지 매칭 → 완료 판정
4. 스텝별 완료 이미지가 없으면 → 사전 검증 단계에서 실행 차단 (OCR 폴백 없음)

### 텔레그램 양방향
- `TelegramPollerThread`: `getUpdates` 5초마다 폴링 (`TELEGRAM_POLL_INTERVAL`)
- 보안 필터: config.json의 `chat_id`와 일치하는 메시지만 처리
- 만료 필터: 메시지 `date`가 30초(`TELEGRAM_CMD_TTL`) 초과면 폐기 — 네트워크 복구 후 밀린 명령 폭주 방지
- 동시성: 스텝 인덱스 변경(`expected_step`/`continue_count`)은 `_step_lock`(`threading.Lock`)으로 보호 → `/next`와 워커 자동 증가의 찰나 경쟁 차단
- `/next` → `force_next_step()` → PAUSED 탈출
- `/screen` → 데몬 스레드에서 전체 화면 캡처+업로드 (GUI 메인 스레드 비차단)

### 양방향 통신 설계 결정 (qasync vs QThread)
v2.0 설계서는 `qasync` + `python-telegram-bot` 단일 루프 통합을 제안했으나,
**기존 `QThread` 폴러 유지**로 결정. 근거:
- 설계서의 핵심 우려(GUI 멈춤)는 별도 폴링 스레드로 이미 해결됨
- qasync 전환은 앱 이벤트 루프 전면 재구성 + 무거운 신규 의존성 2개 + 모든
  블로킹 호출(`pyautogui`/`easyocr`)의 executor 이관을 강요 — 동작 이득 없이 위험만 큼
- 설계서의 실질 가치(/screen·만료 폐기·상태 락)는 루프 구조와 무관하게 구현 가능
  → 셋 다 기존 구조에 반영 완료. `asyncio.Lock` 대신 교차 스레드 등가물 `threading.Lock` 사용.
